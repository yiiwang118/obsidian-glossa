Glossa 上下文、压缩与工具链优化报告

本报告记录本轮针对语言判断、上下文连续性、历史压缩、网页检索和文件下载链路的设计与验证。Markdown 是事实来源，HTML 是由 render-html 生成的阅读视图。

结论

本轮解决了两个直接影响可用性的核心问题：英文 PDF 或选区不再覆盖用户当前的中文表达意图；仅给出论文标题时，下载工具可以先检索候选、核对标题身份、验证响应和文件内容，再写入用户指定目录。

实现同时收紧了历史上下文和界面渲染：旧工具结果不再整段重复发送，模型还可以主动裁剪已经失效的成功工具证据，而可见聊天与审计记录保持完整；长会话首次只渲染最近 6 轮，较早内容按页加载。长文本支持精确行区间，多个已知文本路径可以单次批量读取。PDF 和图片读取增加并发合并、指纹失效与有界内存缓存，PDF 文本提取消除了逐页重复拼接，扫描型 PDF 可按需渲染最多 4 页视觉证据。Skill 系统增加创建工作流、结构校验和工具引用校验。当前生产构建通过全量测试、严格类型与 lint、依赖审计、构建检查和 Obsidian 审查扫描。

问题与根因

语言判断被附件语言劫持

原场景中，用户输入 /explain BLIP2是什么方法 并附带英文 PDF。模型先用英文回答；用户再说“用中文”时，系统又把仍然可见的 PDF 选区当成新任务，反问用户想对文档做什么。

根因不是单一的语言检测错误，而是三个优先级混在了一起：

  1. 用户当前消息的回答语言。
  2. 附件或选区的源语言。
  3. 当前任务是继续、修订还是新建。

旧逻辑缺少明确的数据边界，英文源材料因字符数更多而压过了简短中文指令；“用中文”又因存在显式选区而没被识别为对上一回答的修订。

下载链路只有 URL，没有检索和身份校验

原下载工具适合“已有直链”的情况，但用户通常只给论文标题和目标目录。检索层依赖单一无密钥来源时，可能遇到结果稀疏、人机验证或只返回落地页；即使找到了相近论文，也缺少题名身份门槛和 PDF 内容验证。

因此失败模式包括：

  • 搜索不到作者或机构镜像。
  • 把关键词相似但不是目标论文的 PDF 当成结果。
  • 把 HTML 拒绝页、登录页或 404 内容写成 .pdf。
  • 把 paper/RL 当成文件名而不是已有目录。

安全边界判断

所有新增能力都限定在 Vault API 和现有网络边界内，没有加入终端执行、任意 JavaScript、系统文件、身份/环境信息或持久机器记忆。批量读取只接受模型已经知道的明确 Vault 路径，不扫描整个库；上下文裁剪只处理成功的历史读取/搜索结果，写入与下载确认、Skill、工具披露、当前计划和失败诊断证据不可裁剪。

当前打开文件默认进入本轮上下文，但不再无条件成为任务对象。提示策略会按用户动作区分角色：总结、解释、翻译、分析等内容操作在没有其他目标时把当前文件作为默认对象；存在显式附件时附件优先；与文件无关的连续任务只把当前文件当背景。Markdown 在发送时刷新最新正文，PDF/图片在发送时懒加载；PDF 会依据提示选择摘要采样、标题检查或广泛读取。用户不需要为了总结当前文件额外选中文本，模型也被明确禁止仅因没有选区而调用 get_selection。图片只在用户明确继续讨论上一张图或任务连续性成立时短期复用，切换会话或出现无关新任务会清空，且 base64 不写入聊天历史。

实现说明

每轮回答语言独立决策

回答语言按以下顺序决定：

  1. 当前消息中的明确指令，例如“用中文”“answer in English”。
  2. 去掉 /explain 等命令前缀后的当前用户自然语言。
  3. 最近用户消息的语言习惯。
  4. UI 语言作为最后回退。

选区、附件和 PDF 的语言只写入来源描述，不参与覆盖回答语言。模型收到的约束类似：

<response-language target="zh" confidence="explicit">
Reply in Chinese. The attached source is English; source language does not determine reply language.
</response-language>

界面上的选区预览增加简短语言标记，例如 中文 或 EN，没有恢复占空间的翻译按钮或图标。

短纠正与任务连续性

“用中文”“更简洁一点”等高置信度短消息会被识别为上一回答的修订。即使选区仍存在，也会把上一条显式用户请求和上一条回答一并提供给模型，并禁止退化成“你想如何处理选区”的泛化询问。

而“翻译选中的内容成中文”包含新的对象和动作，不会被误判为只修改上一回答的风格。

上下文清单与历史裁剪

每个上下文项现在带有：

  • role: 用户显式附加，或环境中的当前上下文。
  • kind: 选区、文件、PDF 等。
  • language: 内容检测结果。
  • source: 路径或标签。

历史发送采用“近处完整、远处摘要”的策略：最近两条带工具调用的 assistant 消息保留完整参数、结果和富媒体；更早的长文本只保留开头、结尾以及包含错误、下载来源、SHA、路径和下一步的关键行。旧图片块不再随每轮重复进入上下文。

可恢复的压缩摘要

压缩摘要固定包含以下章节：

  • 当前任务与目标。
  • 用户纠正与回答偏好。
  • 关键实体和证据。
  • 尝试、失败和决策。
  • 已完成与待处理。
  • 最近用户消息原文。
  • 下一步行动。

斜杠命令保存用户看到的原始输入，而不是内部展开后的模板。选区路径、失败工具的具体错误、URL、DOI、标题和文件路径会被保留。转录过长时，算法优先保存初始目标、既有压缩摘要和最近尾部，而不是简单从头截断。

媒体处理与长会话效率

PDF 文本提取由逐页执行 join 的重复拼接改为累计字符计数，复杂度由随页数反复复制的大开销路径收敛为线性累计；每 4 页主动让出主线程，降低大文档读取时的界面卡顿。相同文件和参数的并发读取共享同一个 Promise，失败项立即移出缓存，后续可以重试；缓存键包含路径、修改时间、大小和处理参数，文件变化后不会返回旧结果。每个调用独立响应自己的取消信号：一个调用停止等待不会连带取消其他调用者所需的共享计算。

扫描件或图表密集 PDF 可由 read_pdf 返回页面视觉块。渲染默认缩放为 1.7，最多 4 页，并限制 1200 万像素；canvas 转为 JPEG 后立即归零释放。OpenAI 兼容接口会在完整的 assistant(tool_calls) -> tool... 组之后追加视觉内容，既保留协议要求的工具消息顺序，也让模型真正看到工具返回的图片；Anthropic 路径继续使用原生 tool_result 富内容。

聊天落盘会移除工具图片 base64、重复预览和仅供当前请求使用的模型绑定结果，只保留文本结果与元数据。内存中的当前会话仍可显示和继续使用图片，但 chats.json 不再因多次图片工具调用持续膨胀。

公式渲染兼容

历史聊天中保存的公式内容完整，故障发生在渲染阶段：Obsidian Markdown 会先把 \(...\) 和 \[...\] 解析为普通转义字符，MathJax 随后无法识别公式边界。统一渲染入口现在会在 Markdown 解析前，将代码块、行内代码和 HTML code/pre 之外的完整反斜杠公式边界转换为 Obsidian 原生的 $...$ 和 $$...$$。转义形式、未闭合形式及代码示例保持原样。

流式输出会识别这两类公式边界，并在公式尚未闭合时暂缓提交不完整尾部；默认系统提示同时要求模型优先输出 Obsidian 原生公式边界。旧聊天无需迁移，使用新构建重新渲染即可恢复。回归测试包含截图中的列表、\mathrm、行内公式和块级公式组合，并覆盖嵌套代码围栏、不同长度反引号、HTML 代码和占位符碰撞。

Skills 与按需工具面

默认发送给模型的任务工具由 31 个压缩为 15 个，并额外保留 1 个轻量上下文控制工具；当前首轮共 16 个 schema、14,780 字符，相比原 21,662 字符减少约 32%。保留在首轮的是读取、精确编辑、计划、媒体、主要网络流程、Skill、工具发现和上下文裁剪；整文件覆盖/创建/删除、批量读取、Skill 校验、标签与属性、反链与出链、周期笔记、Canvas、编辑器导航、原始网页搜索和插件桥改为按需加载。已弃用的 edit_section、append_to_note、discover_skills 和 run_skill 继续兼容历史调用，但不再向模型展示。

旧版 tool_search 只把 schema 写进普通工具结果，下一轮 provider 请求仍没有对应函数定义，模型实际上无法可靠调用。本轮为工具结果增加结构化 loadedToolNames，agent loop 会在权限、运行模式、Provider 限制和插件桥状态过滤后，把通过的完整 ToolSpec 加入下一轮请求。搜索支持工具名、别名、能力标签、描述、参数名和常见中文能力词；成功后只返回短说明，避免在聊天历史中重复一份 schema JSON。Skill 也可以声明 required-tools，调用一次后通过同一机制自动加载专用工具。

所有本地工具参数在审批和执行前统一校验 required、类型、枚举、长度/数值/数组范围及未知顶层字段。注册表审计要求每个工具名称、描述、必填字段、参数说明、搜索元数据和 additionalProperties: false 保持一致。当前审计结果为 0 issue。

内置 Skill 重写为明确的“目标、工作流、约束、完成标准”结构：obsidian-markdown、obsidian-canvas、obsidian-bases、pdf-analysis、image-analysis 和 skill-creator。Canvas 工作流使用 read_canvas/patch_canvas 并验证 ID 与边端点；PDF 工作流按任务和页面取证；图片工作流区分整图、裁剪、OCR、图表和像素采样。Skill Creator 先定义正向与负向触发样例、选择约束强度，再创建 SKILL.md 并调用 validate_skill。校验器检查 kebab-case 名称、发现描述、工作流/完成标准、Vault 相对路径和实际可用工具。Skill 文件在创建、修改、删除或重命名后都会立即使发现缓存失效。Skill 列表预算由 1% 收紧到 0.5%，设置 512–6000 字符硬边界，并优先保留名称，防止大量 Skill 挤占上下文。

读取工具也按相同原则分层：高频 read_note 直接支持 1-based 的 start_line、end_line 和 max_lines，返回准确的继续读取位置；低频 read_files 通过 tool_search 加载，对最多 8 个明确文本路径并行读取并隔离单文件失败，批次总输出固定封顶。PDF 和图片不会被误当文本，而是明确引导到 read_pdf 和 view_image。工具返回的 Error: 现在会进入真正的错误状态并传给 provider，而不再以“成功文本”掩盖失败。

context_prune 的裁剪集合由运行时重新验证：只接受已经成功完成、无副作用的读取/搜索 tool call ID。写入、下载和控制类结果受保护。过滤发生在 provider 请求副本上，原消息数组、UI 卡片和落盘审计不变；若一个 assistant 消息包含多个工具调用，只移除被接受的调用及对应结果，继续保持函数调用协议配对。裁剪结果本身会记录接受与忽略的 ID，因此下一轮可以恢复同一状态。

从标题到可信文件的下载流水线

用户标题 + save_to
        |
        v
OpenAlex + arXiv + DuckDuckGo HTML + Yahoo HTML
        |
        v
URL 去重与排名 -> 论文标题身份分数 >= 0.72
        |
        v
逐个拉取候选 -> 2xx -> 未截断 -> 类型允许 -> PDF magic bytes
        |
        v
确认目标目录和文件名 -> 写入 Vault -> 返回来源、大小、SHA256

学术标题包含至少四个有效词且没有直接 DOI 时，候选必须通过题名身份门槛。这样可以拒绝“Policy search for motor primitives in robotics”这类主题接近但身份不同的论文。

下载发生在所有校验之后。PDF 必须以 %PDF- 开头；HTML 落地页、访问拒绝页、404、截断响应和空响应都不会写入 Vault。save_to 指向已有 TFolder 时，即使没有尾部斜杠也会按目录处理。

网络读取优先使用 Obsidian 的 requestUrl，流式文件获取在浏览器 fetch 失败时有 Node HTTP(S) 传输回退；实现没有读取系统身份、环境变量，没有直接使用 fs，也没有执行 shell。

审查边界

当前证据能证明本轮代码通过仓库内配置的 Obsidian 源码审查规则和行为扫描，不会新增已知的 fs、child_process、系统身份或环境变量访问，也没有新增 lint warning。它不能替代 Obsidian 官方目录服务未来规则变化后的远端审核，因此发布前仍应以同一提交重新运行 npm run check，并核对官方 CI 结果。
